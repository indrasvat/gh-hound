Hunt down your GitHub Actions CI

The gh hound --version banner — ANSI Shadow figlet, Bramble green→steel→bone gradient. Shown on --version and first-run, exactly as gh ghent does.

01

One job: GitHub Actions

No PRs, no reviews, no fleet sprawl. hound answers the five questions a CI tool must answer instantly — is it green? what's running? why did it fail? re-run it. watch it. — and every screen below maps to a real Actions endpoint that answers exactly one of them.

The name is the brief: a hound picks up the scent of a failing run and tracks it to the root cause, fast. The logo says the same thing — an h that resolves into a hound's head, a terminal-prompt > for the eye, and a live green nose for "on the scent."

The wedge

No mature "k9s for Actions" exists. gh-dash is PR-first; dedicated TUIs are hobby-grade. The CI niche is open.

The moat

De-noised failure view, run-diff, and a dual human/agent surface — the parts the web UI and raw MCP servers won't casually build.

The discipline

v1 ships only the daily loop. No billing, no charts, no fleet until the loop is something you can't work without.

02

What's broken in the Actions web UI

These are the most-cited, real complaints from engineers debugging Actions in 2026 — and exactly how hound answers each. The right column is not aspiration; it's the v1 feature list, reframed as relief.

The DMV of CI

Checks summary → run → job → step → logs is three or four clicks, each a fresh page with its own spinner. "You are navigating a bureaucracy."
— iankduncan.com, Feb 2026

One launch, zero clicks

gh hound opens straight to your branch's runs with the newest in focus and the failed step pre-selected. Drill is one keypress, back is one keypress — no page loads.

Logs reveal slowly · browser hitches

"The logs appear slowly, like a manuscript revealed one line at a time." The web log viewer is heavy enough to crash a browser tab.
— iankduncan.com, Feb 2026

Native terminal rendering

Logs are plain text rendered in a viewport — instant scroll, ANSI intact, search built in. No DOM, no spinner, no crash.

The error hides in nested folds

"Which of the seventeen nested collapsible groups contains the actual error?" The failure is buried in thousands of lines.
— iankduncan.com · gradle.com, 2026

Jump to the failure

The failure view opens on the failed step's error window — ~12 lines around the first error — with annotations surfaced inline. ::group:: folds are navigable, not a maze.

Get the raw log · open ten tabs

"It's slow; you get the raw log; it's large; the failure is in the middle; you scroll up and down; you open plenty of tabs."
— github-commando README

Pipe it · de-noised

One tool, no tabs. gh hound … --no-tui --json returns the failed step + excerpt + annotations as structured data — for you or your agent.

Re-run blind, burn minutes

"The impulse is to re-run and hope. But if you don't know why it failed, you're just burning compute minutes."
— bswen, Apr 2026

Understand, then re-run

See the cause first, then r re-run the job, R re-run only failed jobs, or ↻ the whole run — each a single endpoint, a single key.

03

Default behaviour — what gh hound does with no arguments

Like lazygit and k9s, it opens to a predictable home — but that home is context-scoped and live, so you effectively see your current CI the moment it launches. It resolves the repo from the git remote and the branch from HEAD, then lands on the runs list scoped to your current branch, newest first, auto-selecting the latest run with its step timeline in the detail pane. If that run is in progress, you're already watching it light up — no keypress.

Why not auto-dive into the running run?

A tool you launch reflexively should open somewhere consistent. Auto-attaching full-screen is great right after a push and annoying when you just wanted to glance. So it's one key (w) or gh hound watch — or opt in with auto_watch = true.

The home is the live view

Not a separate "tracking" mode: the runs list polls on an adaptive interval with ETag caching, so the newest run's status and the step timeline update in place. Glance, and you already know.

Fallbacks — the empty-screen failure modes, handled

SituationBehaviour
Branch has zero runs (fresh, never pushed)Auto-widen to repo-wide recent runs with a one-line notice: no runs on fix/parser — showing all branches
Repo has no workflows / Actions disabledExplicit empty state with next steps — never a blank panel
Not in a git repo / no resolvable remoteFriendly error suggesting gh hound -R owner/repo
Detached HEAD / amended-unpushed SHAFall back to branch, then repo-wide — never error out silently

Overrides & config

Mental model in one line: it always opens to a known home — yours just happens to be "my branch's CI, live, newest in focus."

04

Grounded in the real API

Validated against the official REST reference (API version 2026-03-10). The entire Actions domain — runs, jobs, steps, logs, re-run, cancel, dispatch — is REST only; GraphQL has no WorkflowRun/Job/Step types and covers only the Checks rollup on a ref. So REST is the backbone; GraphQL is an optional one-shot "is this ref green" that also catches third-party checks.

Read & logs

GET…/actions/jobs/{job_id}/logsPlain text (302, 60s expiry) — single-job drill-down skips the zip.

05

Performance — busy fast

Taken from GitHub's own rate-limit and best-practice docs, then engineered around.

Authenticated ceiling

The gh token gives 5,000 req/hr vs 60 unauthenticated. (Validation literally hit the 60 wall — proof the auth path is mandatory.)

ETag / 304 cache

Every poll carries If-None-Match. A 304 Not Modified costs zero against the primary limit — idle polling is effectively free.

Adaptive polling

No live-log socket exists. Poll the jobs endpoint ~2–3s while anything is in_progress; back off hard when idle. Step status goes live; logs land on completion.

Serial & spaced

Requests queued serially (secondary-limit safe); mutations spaced ≥1s. The cache is the source of truth; the network never blocks a keystroke.

Decoupled paint

Borrowing vicaya's daemon model: a background poller writes to cache; the render loop reads cache. 60fps feel regardless of latency.

Synchronized output

Bubble Tea v2 uses terminal mode 2026 (synchronized updates) — repaints land atomically, so live timelines never tear or flicker.

10

Progressive disclosure

A terminal has no room to waste. hound answers the 80% question on one line, then reveals depth on demand — four levels, each one ⏎ deeper, ⎋ shallower. A breadcrumb always shows where you are; on narrow terminals the right detail pane collapses into a full-screen push rather than truncating. The most likely next action is always the most prominent thing on screen — a failed run surfaces "rerun failed," a running one surfaces "watch," a green one gets out of your way.

L0
Glance

One header line: branch, counts — "1 failing · 1 running". Answers is it green? before you read a table.

always visible
L1
List

The runs table — status, workflow, duration sparkline, age. The landing screen, and where 90% of time is spent.

⏎ from L0 · the default
L2
Detail

Jobs + step timeline for one run, master-detail. The failed step pre-flagged so the next keystroke is obvious.

⏎ on a run · ⎋ back
L3
Deep

The de-noised error window + annotations — and only if you ask, the full folded log. The long tail stays hidden until summoned.

n jump · l full log · ⤓ expand

hound · gh hound — design spec, grounded in the live API and the shipping Charm v2 stack.

REST schemas, enums, and log shapes validated against docs.github.com/en/rest/actions at API version 2026-03-10; Actions runs/jobs/logs are REST-only, GraphQL covers the Checks rollup. Performance model from GitHub REST best-practices (5,000 req/hr authenticated, ETag→304 free, serial calls, ≥1s between mutations; the unauthenticated 60/hr ceiling was hit during validation).
